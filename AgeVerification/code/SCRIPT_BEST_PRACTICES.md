# Script Best Practices

Three key project-specific guidelines for writing scripts:

## 1. Use Functions, Not Classes

Write simple functions that operate on standard objects like DataFrames:

```python
def load_data(file_path):
    """Load data from file."""
    return pd.read_csv(file_path)

def clean_data(df):
    """Clean the dataframe."""
    # cleaning logic
    return df_clean
```

Functions are easier to understand, test, and modify than classes.

## 2. Mirror Directory Structure for Outputs

When a script in `code/` generates outputs, mirror the **full path** in output directories:

**Script location:**
```
code/ProcessComscore/data_structure_validation/validate_demographic_panel.py
```

**Output locations:**
```
data/ProcessComscore/data_structure_validation/    (CSV files, validation reports, etc.)
output/ProcessComscore/data_structure_validation/  (plots, figures)
```

This makes it easy to find all outputs generated by any script.

## 3. Always Use House Plotting Style

For all plots, import and apply the house style:

```python
from plot_style import apply_plot_style, UCHICAGO_MAROON

apply_plot_style()

# Use UChicago maroon for plots
plt.bar(x, y, color=UCHICAGO_MAROON)
```

This ensures consistent, professional-looking figures with:
- UChicago maroon (#800000) as base color
- Large, readable fonts
- Sparse gridlines

## 4. Reference Issues in Git Commits

Always reference the GitHub issue number in commit messages:

```bash
git commit -m "Add demographic panel validation for issue #8

Detailed description of what was done...

Co-Authored-By: Claude Sonnet 4.5 <noreply@anthropic.com>"
```

Use format: `"Brief description for issue #X"` or `"Brief description #X"`

## 5. Commit Only Meaningful Output Files

**DO commit:**
- Small data files meant for browsing (CSVs < 100KB, summary tables)
- Plots and visualizations (PNG files)
- Analysis scripts and code
- Documentation and reports

**DON'T commit:**
- Large raw data files
- Intermediate processing files
- Temporary outputs
- Files already tracked in raw/ directory

## 6. Always Run Scripts from Terminal

Scripts should always be run from the command line in the project root directory.
Comment out any hardcoded local directory paths (used for IDE testing only):

```python

```

The bash master script will set the correct working directory.

## 8. Build Paths Dynamically from Script Location for Cluster Compatibility

Always construct paths relative to the script's location using `__file__`. This pattern ensures scripts run on both Windows and the Linux cluster.

**Example pattern (adapt paths to your specific script):**

```python
import os

# Get the project root directory (adjust number of ".." based on script depth)
file_dir = os.path.dirname(os.path.abspath(__file__))
project_root = os.path.abspath(os.path.join(file_dir, "..", "..", ".."))

# Build all paths using os.path.join() from project root
# Replace these example paths with your actual file paths
dma_file = os.path.join(project_root, "data", "ProcessAuxiliary", "DMA_County", "DMA_summary.csv")
output_file = os.path.join(project_root, "output", "ProcessComscore", "results.csv")
```

**Note:** The number of `".."` depends on how deep your script is in the directory structure.

## 9. Handle Both .txt and .txt.gz File Extensions

Raw data files are compressed (.txt.gz) on the cluster but uncompressed (.txt) locally. Always check for both versions.

**Example pattern (use your actual file paths):**

```python
# Check for both .txt (local) and .txt.gz (cluster) versions
# Replace with your actual file path
desktop_file_base = os.path.join(project_root, "raw", "desktop_demographics", "US_comscore_machine_demos_202201.txt")
if os.path.exists(desktop_file_base):
    desktop_file = desktop_file_base
elif os.path.exists(desktop_file_base + ".gz"):
    desktop_file = desktop_file_base + ".gz"
else:
    raise FileNotFoundError(f"Could not find file at {desktop_file_base} or {desktop_file_base}.gz")
```

Pandas will automatically decompress .gz files when reading.

## 10. Use Raw Strings for Regex Patterns

When defining regex patterns, use raw strings `r""` to avoid issues with backslash escape sequences.

**Examples:**

```python
# Good - raw string for regex
directional_split_pattern = r'-(?:N|S|E|W|C)$'
pattern = r'\d{5}'  # matches 5 digits

# Bad - regular string (backslashes need double escaping)
pattern = '\\d{5}'
```

## 11. Ensure Cross-Platform Compatibility for Mac and PC Testing

Scripts must work on both Windows (PC) and macOS (Mac) without modification. Follow these guidelines:

**Path handling (see also #8):**
- Always use `os.path.join()` for paths (never hardcode `/` or `\` separators)
- Use `os.path.abspath(__file__)` to get script location

**Avoid platform-specific code:**
```python
# Good - cross-platform
import subprocess
subprocess.run(['python', 'script.py'])

# Bad - Windows-only
os.system('dir')  # use os.listdir() or pathlib instead

# Bad - Mac/Linux-only
os.system('ls')   # use os.listdir() or pathlib instead
```

**Line endings:**
- Use `encoding='utf-8'` when reading/writing files
- All shell scripts (`.sh`, `.bash`) and Python files (`.py`) must use Unix line endings (LF, not CRLF)
- Git automatically handles this via `.gitattributes` - files are stored as LF in the repository
- Scripts with Windows line endings (CRLF) will fail on the Linux cluster with errors like `/bin/bash^M: bad interpreter`

**Test on both platforms:**
- Before committing, test scripts on both Mac and PC when possible
- Document any unavoidable platform-specific requirements in script header
